# 검색 평가 지표: P@k, R@k, MRR@k, nDCG@k

BM25 계열 검색 결과를 qrels 기준으로 평가하는 방법과 실행 코드를 정리한다.

- `evaluate_metrics_bm25.py`: BM25, BM25+, BM25L, BM25F의 검색 품질을 비교하는 CLI 스크립트

`@k`는 검색 결과의 **상위 k개까지만 평가한다**는 뜻이다. 예를 들어 `P@5`는 상위 5개 검색 결과만 사용해 계산한다.

## 지표 요약

| 지표 | 평가 대상 | 순서 반영 | 적합한 질문 |
| --- | --- | --- | --- |
| P@k | 상위 k개 중 정답의 비율 | X | 상위 결과가 얼마나 정확한가? |
| R@k | 전체 정답 중 상위 k개에서 찾은 비율 | X | 정답을 얼마나 놓치지 않았는가? |
| MRR@k | 첫 번째 정답의 순위 | O | 첫 정답이 얼마나 빨리 나오는가? |
| nDCG@k | 관련도와 순위를 반영한 전체 랭킹 | O | 중요한 정답을 위에 잘 배치했는가? |

## 1. P@k (Precision at k)

상위 k개 검색 결과 중 **정답 문서가 차지하는 비율**이다.

$$
P@k = \frac{\text{상위 } k \text{개 안의 정답 문서 수}}{k}
$$

상위 5개 결과가 `정답, 오답, 정답, 오답, 오답`이라면 정답이 2개이므로 다음과 같다.

$$
P@5 = \frac{2}{5} = 0.4
$$

### 특징

- 상위 결과의 정확도를 평가한다.
- 정답을 얼마나 많이 찾았는지보다 사용자에게 보여 준 결과가 얼마나 깔끔한지에 초점을 둔다.
- 정답 개수만 세므로 정답이 1위인지 5위인지는 반영하지 않는다.

## 2. R@k (Recall at k)

전체 정답 문서 중 **상위 k개 안에서 찾아낸 문서의 비율**이다.

$$
R@k = \frac{\text{상위 } k \text{개 안의 정답 문서 수}}{\text{전체 정답 문서 수}}
$$

전체 정답이 4개이고 상위 5개 결과에서 그중 2개를 찾았다면 다음과 같다.

$$
R@5 = \frac{2}{4} = 0.5
$$

### 특징과 주의점

- 전체 정답을 얼마나 놓치지 않고 회수했는지 평가한다.
- 정답 수가 많은 데이터에서 특히 중요하다.
- 정답의 순서는 반영하지 않는다.
- 전체 정답 수에 비해 k가 너무 작으면 가능한 최대 점수도 낮아진다.

예를 들어 전체 정답이 100개인데 `R@5`를 평가하면, 상위 5개를 모두 맞혀도 최대값은 다음과 같다.

$$
R@5_{\text{ max}} = \frac{5}{100} = 0.05
$$

따라서 1차 retriever의 회수 능력을 평가할 때는 `Recall@10`, `Recall@50`, `Recall@100`처럼 더 큰 k도 함께 확인하는 경우가 많다.

## 3. MRR@k (Mean Reciprocal Rank at k)

첫 번째 정답 문서가 **얼마나 빠른 순위에 등장했는지** 평가한다.

### Reciprocal Rank (RR)

한 query에서 첫 번째 정답이 나온 순위를 `rank`라고 하면 다음과 같다.

$$
RR = \frac{1}{rank}
$$

- 첫 정답이 1위: $1/1 = 1.0$
- 첫 정답이 2위: $1/2 = 0.5$
- 첫 정답이 5위: $1/5 = 0.2$
- 상위 k개 안에 정답이 없음: $0$

### Mean Reciprocal Rank (MRR)

여러 query에서 계산한 RR의 평균이다.

$$
MRR = \frac{1}{|Q|} \sum_{q \in Q} \frac{1}{rank_q}
$$

- $Q$: 전체 query 집합
- $rank_q$: query $q$에서 첫 정답이 나온 순위

세 query의 첫 정답 위치가 각각 1위, 2위, 4위라면 다음과 같다.

$$
MRR = \frac{1 + \frac{1}{2} + \frac{1}{4}}{3}
= \frac{1.75}{3}
\approx 0.583
$$

### 특징

- 첫 번째 정답만 평가에 사용한다.
- 두 번째 이후의 정답 위치는 반영하지 않는다.
- FAQ나 QA 검색처럼 정답 하나를 빠르게 찾는 문제가 잘 맞는다.
- 다른 지표도 여러 query에 대한 평균을 사용하지만, 관습적으로 MRR에만 `Mean`을 명시한다.

## 4. nDCG@k (Normalized Discounted Cumulative Gain at k)

nDCG@k는 관련도 높은 정답을 상위에 배치할수록 높은 점수를 주는 지표다. 정답/오답뿐 아니라 문서별 관련도 차이도 반영할 수 있다.

### P@k, R@k만으로 부족한 이유

다음 두 결과는 모두 상위 5개 중 정답이 2개이므로 `P@5`가 같다.

- 경우 A: `정답, 정답, 오답, 오답, 오답`
- 경우 B: `오답, 오답, 오답, 정답, 정답`

사용자는 보통 위에서부터 결과를 확인하므로 실제 검색 품질은 A가 더 좋다. DCG와 nDCG는 이런 **정답의 위치**를 반영한다.

### DCG

DCG(Discounted Cumulative Gain)는 뒤쪽 결과의 기여도를 로그로 할인한다. 이 프로젝트의 구현은 관련도에 exponential gain을 적용한다.

$$
DCG@k = \sum_{i=1}^{k} \frac{2^{rel_i} - 1}{\log_2(i+1)}
$$

- $i$: 검색 결과의 순위
- $rel_i$: 해당 순위 문서의 관련도 점수

관련도는 예를 들어 다음처럼 부여할 수 있다.

- `3`: 매우 관련 있음
- `2`: 꽤 관련 있음
- `1`: 조금 관련 있음
- `0`: 관련 없음

로그를 사용하면 순위가 내려갈수록 기여도가 줄어들지만, 단순히 순위 $i$로 나누는 것보다 완만하게 할인된다. binary relevance처럼 모든 정답의 관련도가 `1`이라면 순위별 gain은 다음과 같다.

- 1위: $1 / \log_2(2) = 1.0$
- 2위: $1 / \log_2(3) \approx 0.63$
- 3위: $1 / \log_2(4) = 0.5$
- 5위: $1 / \log_2(6) \approx 0.39$

### DCG 계산 예시

상위 5개 결과의 관련도가 `[3, 2, 0, 1, 0]`이라고 하자.

$$
DCG@5 =
\frac{2^3-1}{\log_2(2)} +
\frac{2^2-1}{\log_2(3)} +
\frac{2^0-1}{\log_2(4)} +
\frac{2^1-1}{\log_2(5)} +
\frac{2^0-1}{\log_2(6)}
$$

각 항을 계산하면 약 `7`, `1.893`, `0`, `0.431`, `0`이므로 다음과 같다.

$$
DCG@5 \approx 9.323
$$

### IDCG (Ideal DCG)

query마다 관련 문서 수와 관련도 분포가 다르므로 DCG 값만으로는 서로 비교하기 어렵다. 가능한 가장 이상적인 순서로 정렬했을 때의 DCG인 `IDCG`를 함께 계산한다.

위 예시 `[3, 2, 0, 1, 0]`의 이상적인 정렬은 `[3, 2, 1, 0, 0]`이다.

$$
IDCG@5 =
\frac{2^3-1}{\log_2(2)} +
\frac{2^2-1}{\log_2(3)} +
\frac{2^1-1}{\log_2(4)}
\approx 9.393
$$

### nDCG

실제 DCG를 이상적인 IDCG로 나누어 query 사이의 점수 범위를 정규화한다.

$$
nDCG@k = \frac{DCG@k}{IDCG@k}
$$

위 예시에서는 다음과 같다.

$$
nDCG@5 = \frac{9.323}{9.393} \approx 0.993
$$

IDCG는 해당 query에서 가능한 최대 DCG이므로 nDCG는 일반적으로 0과 1 사이가 된다.

- `1`에 가까울수록 이상적인 정렬에 가깝다.
- `0`에 가까울수록 랭킹 품질이 낮다.

### 특징과 한계

- 정답의 위치와 관련도 등급(graded relevance)을 모두 반영한다.
- 정답이 여러 개이고 중요도 차이가 있는 검색 문제에 적합하다.
- retriever와 reranker의 전체 랭킹 품질을 비교할 때 유용하다.
- 상위 k개 밖의 고품질 정답은 반영하지 못하므로 전체 회수 능력은 Recall@k와 함께 확인해야 한다.
- 상위 결과에서 정답을 거의 회수하지 못하면 재정렬 품질을 충분히 평가하기 어렵다. 먼저 Recall@k를 확인하는 것이 좋다.

## 지표 선택 기준

- `Recall@k`: 정답을 후보군 안에 충분히 넣었는지 확인한다.
- `MRR@k`: 첫 정답이 얼마나 빨리 나오는지 확인한다.
- `nDCG@k`: 관련도 높은 정답을 위에 잘 배치했는지 확인한다.
- `P@k`: 사용자에게 보여 준 상위 결과가 얼마나 정확한지 확인한다.

하나의 지표만으로 검색 품질 전체를 설명하기 어렵기 때문에 검색 단계와 사용 목적에 맞춰 여러 지표를 함께 보는 것이 좋다.

## 실행 방법

모든 명령은 repository 루트(`/home/test0000/study-notes`)에서 실행한다.

### 기본 실행

```bash
python3 02-search-ir/evaluation/metrics/evaluate_metrics_bm25.py
```

기본값은 BM25, BM25+, BM25L, BM25F 모델을 모두 `top-k=10` 기준으로 평가한다.

### 특정 모델만 평가

```bash
python3 02-search-ir/evaluation/metrics/evaluate_metrics_bm25.py \
  --model bm25f \
  --top-k 10
```

`--model`에는 `all`, `bm25`, `bm25+`, `bm25l`, `bm25f`를 지정할 수 있다.

### 데이터 파일 직접 지정

```bash
python3 02-search-ir/evaluation/metrics/evaluate_metrics_bm25.py \
  --docs 02-search-ir/bm25/data/generated/corpus.jsonl \
  --queries 02-search-ir/bm25/data/generated/queries.json \
  --qrels 02-search-ir/bm25/data/generated/qrels.json \
  --top-k 10
```

`corpus.jsonl`, `queries.json`, `qrels.json`이 없으면 기존 `bm25f_*` generated 파일을 fallback으로 사용한다.

스크립트는 각 query의 지표를 계산한 뒤 모델별 평균 `P@k`, `R@k`, `MRR@k`, `nDCG@k`를 출력한다.
