# BM25, BM25+, BM25L, BM25F 정리

검색 시스템에서 query와 문서가 얼마나 관련 있는지 점수를 계산할 때 자주 사용하는 대표적인 랭킹 함수들이다.

이 폴더의 `bm25.py`에서는 아래 4가지를 구현하고 있다.

- BM25
- BM25+
- BM25L
- BM25F

핵심은 모두 **query에 들어 있는 단어가 문서에 얼마나 잘 등장하는지**를 보는 방식이다.  
다만 문서 길이, 단어 반복, 필드 가중치를 어떻게 처리하느냐가 조금씩 다르다.

---

## 1. BM25

BM25는 정보 검색에서 가장 널리 쓰이는 sparse retrieval 랭킹 함수 중 하나다.

기본 아이디어는 아래와 같다.

- query 단어가 문서에 있으면 점수가 올라간다.
- 흔한 단어보다 드문 단어가 더 높은 점수를 준다.
- 단어가 많이 반복될수록 점수는 올라가지만, 무한정 커지지는 않는다.
- 문서가 너무 길면 단어가 우연히 많이 들어갈 수 있으므로 길이 정규화를 한다.

---

### BM25에서 보는 값

#### 1) TF

`TF`는 term frequency다.

즉, 특정 단어가 문서 안에 몇 번 등장했는지를 의미한다.

예를 들어 query가 `bm25 검색`이고 어떤 문서에 `bm25`가 3번 등장하면, `bm25`의 TF는 3이다.

하지만 BM25는 TF가 커진다고 점수를 계속 크게 올리지 않는다.  
어느 정도 이상 반복되면 점수 증가폭이 줄어든다.

이걸 **TF saturation**이라고 볼 수 있다.

---

#### 2) IDF

`IDF`는 inverse document frequency다.

전체 문서 중에서 해당 단어가 얼마나 드문지를 나타낸다.

- 많은 문서에 나오는 단어: 낮은 IDF
- 적은 문서에만 나오는 단어: 높은 IDF

예를 들어 검색 데이터에서 `문서`, `검색` 같은 단어는 자주 나올 수 있다.  
반면 `BM25F`, `nDCG` 같은 단어는 상대적으로 적게 나올 수 있다.

드문 단어일수록 검색 의도를 더 잘 설명할 가능성이 높기 때문에 더 큰 점수를 준다.

---

#### 3) 문서 길이 정규화

긴 문서는 단어가 많이 들어갈 가능성이 높다.

그래서 단순히 단어 등장 횟수만 보면 긴 문서가 과하게 유리해질 수 있다.

BM25는 평균 문서 길이와 현재 문서 길이를 비교해서 점수를 보정한다.

---

### 코드에서의 BM25

`bm25.py`에서는 `bm25()` 함수가 기본 BM25를 계산한다.

```python
def bm25(
    query_tokens: list[str],
    index: FlatIndex,
    *,
    k1: float = 1.2,
    b: float = 0.75,
) -> dict[str, float]:
```

여기서 중요한 파라미터는 `k1`, `b`다.

- `k1`: 단어 반복 횟수의 포화 정도를 조절
- `b`: 문서 길이 정규화 강도를 조절

`b`가 0이면 문서 길이를 거의 보지 않는다.  
`b`가 1에 가까우면 문서 길이를 더 강하게 반영한다.

---

## 2. BM25+

BM25+는 기본 BM25의 길이 정규화 문제를 완화하기 위한 변형이다.

기본 BM25는 긴 문서에 대해 점수가 너무 낮게 나오는 경우가 있다.  
특히 query 단어가 실제로 포함되어 있는데도, 문서가 길다는 이유로 과하게 불리해질 수 있다.

BM25+는 여기에 `delta` 값을 더해서 이런 문제를 줄인다.

---

### BM25+의 핵심

BM25+는 TF component에 일정한 보정값을 더한다.

```python
tf_component + delta
```

이렇게 하면 query 단어가 등장한 문서가 길이 때문에 지나치게 낮은 점수를 받는 상황을 줄일 수 있다.

---

### 코드에서의 BM25+

`bm25.py`에서는 `bm25_plus()` 함수가 BM25+를 계산한다.

```python
def bm25_plus(
    query_tokens: list[str],
    index: FlatIndex,
    *,
    k1: float = 1.2,
    b: float = 0.75,
    delta: float = 1.0,
) -> dict[str, float]:
```

추가된 파라미터는 `delta`다.

- `delta`: 긴 문서가 과하게 불리해지는 현상을 줄이기 위한 보정값

---

### BM25+가 유리한 경우

- 문서 길이 차이가 큰 corpus
- 긴 문서 안에 query 단어가 의미 있게 포함되어 있는 경우
- 기본 BM25가 긴 문서를 너무 강하게 패널티 준다고 느껴지는 경우

---

## 3. BM25L

BM25L도 BM25의 길이 정규화 문제를 완화하기 위한 변형이다.

BM25+가 TF component에 `delta`를 더하는 방식이라면,  
BM25L은 **길이 정규화가 적용된 TF**에 `delta`를 더한다.

즉, 보정이 들어가는 위치가 다르다.

---

### BM25L의 핵심

먼저 문서 길이를 반영해서 TF를 정규화한다.

```python
normalized_tf = tf / length_norm
```

그다음 정규화된 TF에 `delta`를 더해서 점수를 계산한다.

```python
normalized_tf + delta
```

이 방식은 긴 문서에 대한 패널티를 조금 더 부드럽게 만드는 데 목적이 있다.

---

### 코드에서의 BM25L

`bm25.py`에서는 `bm25l()` 함수가 BM25L을 계산한다.

```python
def bm25l(
    query_tokens: list[str],
    index: FlatIndex,
    *,
    k1: float = 1.2,
    b: float = 0.75,
    delta: float = 0.5,
) -> dict[str, float]:
```

BM25+와 마찬가지로 `delta`가 있지만 기본값은 다르다.

- BM25+: `delta = 1.0`
- BM25L: `delta = 0.5`

이 값은 절대적인 정답이 아니라 실험하면서 조정할 수 있는 파라미터다.

---

### BM25L이 유리한 경우

- 긴 문서와 짧은 문서가 섞여 있는 corpus
- 길이 정규화가 필요하지만 기본 BM25보다 완만한 보정을 원할 때
- BM25+보다 조금 다른 방식으로 긴 문서 패널티를 조정하고 싶을 때

---

## 4. BM25F

BM25F는 문서를 하나의 본문으로만 보지 않고, 여러 필드로 나누어 점수를 계산하는 방식이다.

예를 들어 이 예제 코드의 문서는 아래 필드를 가진다.

- `title`
- `tags`
- `text`

일반 BM25 계열은 이 필드들을 하나의 긴 텍스트처럼 합쳐서 본다.  
반면 BM25F는 필드별로 따로 보고, 각 필드에 다른 가중치를 줄 수 있다.

---

### 왜 필드가 중요할까?

같은 단어라도 어디에 등장했는지에 따라 중요도가 다를 수 있다.

예를 들어 query가 `bm25 검색 랭킹`이라고 하자.

어떤 문서의 제목이 아래와 같다면:

```text
BM25 검색 랭킹 기본 개념
```

이 문서는 query와 매우 관련 있어 보인다.

반면 본문 어딘가에 `bm25`라는 단어가 한 번만 등장했다면, 제목에 등장한 것보다는 중요도가 낮을 수 있다.

그래서 BM25F는 필드별 가중치를 줄 수 있다.

---

### 코드에서의 BM25F

`bm25.py`에서는 `bm25f()` 함수가 BM25F를 계산한다.

```python
def bm25f(
    query_tokens: list[str],
    index: FieldIndex,
    *,
    k1: float = 1.2,
    field_weights: dict[str, float] | None = None,
    field_b: dict[str, float] | None = None,
) -> dict[str, float]:
```

현재 기본 field weight는 아래와 같다.

```python
weights = {
    "title": 2.0,
    "tags": 2.0,
    "text": 0.5,
}
```

즉, 이 예제에서는 `title`, `tags`에 등장한 query 단어를 본문보다 더 중요하게 본다.

필드별 길이 정규화 값은 아래와 같다.

```python
b_values = {
    "title": 0.5,
    "tags": 0.0,
    "text": 0.75,
}
```

- `title`: 어느 정도 길이 정규화
- `tags`: 길이 정규화 거의 없음
- `text`: 본문 길이 정규화를 강하게 적용

---

### BM25F가 유리한 경우

- 문서가 `title`, `body`, `tags`, `category`처럼 구조화되어 있을 때
- 제목이나 태그에 나온 단어를 더 중요하게 보고 싶을 때
- 일반 BM25보다 검색 도메인에 맞는 랭킹 조정이 필요할 때

---

## 4가지 방식 비교

| 모델 | 핵심 특징 | 언제 유리한가 |
| --- | --- | --- |
| BM25 | 기본 sparse retrieval 랭킹 함수 | 일반적인 키워드 검색 |
| BM25+ | TF component에 delta 보정 추가 | 긴 문서가 과하게 불리할 때 |
| BM25L | 정규화된 TF에 delta 보정 추가 | 길이 보정을 더 부드럽게 하고 싶을 때 |
| BM25F | 필드별 가중치와 필드별 길이 정규화 적용 | title, tags, text처럼 필드가 나뉜 문서 |

---

## 이 예제 코드에서의 흐름

검색은 대략 아래 순서로 동작한다.

1. 문서와 query를 로드한다.
2. query와 문서를 토큰화한다.
3. 문서들로 역색인을 만든다.
4. 선택한 BM25 모델로 문서별 점수를 계산한다.
5. 점수가 높은 문서를 상위 결과로 출력한다.

실행 예시는 아래와 같다.

```bash
python3 02-search-ir/bm25/bm25.py --model bm25f --query "bm25 검색 랭킹" --top-k 10
```

---

## 정리

BM25 계열은 모두 키워드 기반 검색 모델이다.

가장 기본은 BM25이고, BM25+와 BM25L은 문서 길이 정규화 문제를 완화하기 위한 변형이다.  
BM25F는 문서가 여러 필드로 나뉘어 있을 때 필드별 중요도를 반영하기 위한 방식이다.

간단히 말하면:

- 기본 키워드 검색이면 `BM25`
- 긴 문서 패널티를 줄이고 싶으면 `BM25+` 또는 `BM25L`
- 제목, 태그, 본문을 다르게 보고 싶으면 `BM25F`

이 예제 데이터는 `title`, `tags`, `text` 필드가 있고 qrels도 title/tags 겹침을 중요하게 보도록 생성되어 있다.  
그래서 현재 실험에서는 BM25F가 다른 모델보다 좋은 점수를 내기 쉽다.
